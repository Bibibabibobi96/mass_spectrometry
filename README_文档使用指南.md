# COMSOL 仿真项目文档使用指南（AI 与人类共用）

> 这份文档是"文档的文档"——告诉你（无论是AI还是人）这个目录下每份经验文档是干什么的、
> 遇到什么情况该查哪一份、新发现该往哪份文档里加、加的时候要遵守什么规则。**每次开始
> 新任务前，先看这份指南决定去读哪份文件，不要每次都从头翻旧的单体大文档。**

## 文档体系总览

本目录下的经验文档分成四类，边界很明确：

| 文件 | 回答什么问题 | 什么时候查/什么时候写 |
|---|---|---|
| `COMSOL_MATLAB_API手册.md` | "这个COMSOL/MATLAB调用该怎么写？这个属性叫什么名字？合法取值是什么？有没有现成验证过的部件可以直接复用？" | 写任何新脚本、建任何新几何/物理场/求解器配置之前先查§7.0部件库表；踩到"某个调用报错/行为诡异"时先查§7.12黑名单和对应小节。**任何项目**（不管是质谱仪哪个部件、电子枪、还是以后的新项目）都可能用到。 |
| `COMSOL_调试方法论.md` | "结果不对/比预期差，但不知道从哪查起，该用什么策略缩小排查范围？"、"服务端卡死/崩溃了怎么恢复？"、"这类几何/选择集问题有没有通用陷阱？" | 遇到"分辨率不达预期"、"场精度有偏差但不知道是谁的问题"、"MATLAB/COMSOL卡死"这类**方法论层面**的困境时查。这里的每一条都是跨项目验证过的策略，不是某个项目的具体参数。 |
| `project_modelB/项目_oaTOF环栈反射镜_ModelB.md` | "这个具体的oa-TOF双级反射镜模型，现在的参数是什么？为什么是这些参数？踩过哪些只在这个几何上才会出现的坑？" | 修改`project_modelB/ms_modelB_ringstack_reflectron.m`或任何`scan_*.m`脚本之前必须先读，了解当前状态和历史教训，避免重复踩坑或推翻已验证的结论。 |
| `project_eGun/项目_电子枪.md` | "电子枪（螺旋灯丝+Wehnelt）项目的物理结论是什么？" | 修改`project_eGun/`下的phase*系列脚本之前查。 |

## 目录结构

```
comsol_scripts/
  CLAUDE.md                          # 自动加载的入口，指向本文件
  README_文档使用指南.md               # 本文件
  COMSOL_MATLAB_API手册.md            # 通用API速查（跨项目）
  COMSOL_调试方法论.md                 # 通用调试方法论（跨项目）
  common/                            # 可复用组件验证脚本（test_*.m），任何项目都可能引用
  project_modelB/                    # Model A(碰撞冷却)+Model B(oa-TOF环栈反射镜)正式脚本+项目文档
  project_eGun/                      # 电子枪 phase*.m 系列脚本+项目文档
```
`comsol_models/`（.mph模型）和`comsol_results/`（图片/数据）是同级的独立目录，**没有**
按项目拆子目录（这两个目录体积大、含二进制文件，拆分收益不明显且风险较高，暂时保持扁平，
靠文件名前缀区分项目——比如`MS_ModelB_*`/`ElectronGun_*`/`Multipole*`）。脚本里引用
`comsol_models`/`comsol_results`的路径都是绝对路径，不受`comsol_scripts`内部子目录调整
影响。

**判断一条新发现该写进哪个文件的经验法则**：
- 换个项目、换个部件还成立吗？→ 成立，是"怎么调用API"层面的 → 写进`API手册.md`。
- 换个项目、换个部件还成立吗？→ 成立，是"怎么排查问题"层面的策略/方法论 → 写进
  `调试方法论.md`。
- 只对这一个具体模型/这一套具体几何参数成立（比如"这个反射镜的d1应该取120mm"、"这个
  电子枪的Wehnelt孔径该多大"）→ 写进对应的`项目_*.md`。
- **同一个主题"表面上像API调用，其实是两种不同性质的知识"时，拆开分别归类**：例如
  画图——"怎么调API让标题变成这样"(`titletype='manual'`这类机械调用写法)属于API手册；
  "标题必须包含什么信息、颜色语义要不要跨图固定"这类**为什么要这么做**(防止review结果
  时被小样本噪声/图与图颜色语义不一致误导)的规范，本质是调试严谨性问题，应该写进
  `调试方法论.md`，不要因为表面上都叫"画图"就塞进同一份文档。判断标准依然是"这条知识
  换个项目是否还成立、成立的原因是API机制还是排查方法论"，跟其他分类判断同一套逻辑。
- 新开一个全新项目（不是给现有项目加功能）→ 新建一个`项目_<项目名>.md`，并在这份指南
  的表格里加一行。

## 各文档内部的维护规则

### `COMSOL_MATLAB_API手册.md`
- §7.0"已验证部件库速查表"和"7内容分类索引"是最高频被查阅的入口，新增一个完整部件
  时，两处都要同步加一行/一条，且要保持指向正确的小节号。
- §7.x的小节编号**只递增，不要在已有编号中间插入**——文档内外都有大量`见§7.x`交叉引用
  依赖编号稳定，插队会破坏引用。
- §7.12黑名单：确认某个调用无效/报错时顺手加一条，节省下次踩同一个坑的时间。
- 每条尽量只留"怎么写是对的"+一句话背景，详细的排查过程如果本身有通用方法论价值，放
  进`调试方法论.md`并互相`见xxx.md`引用；如果只是这一个项目的踩坑细节，放进对应的
  `项目_*.md`，这份API手册里不要堆砌叙事。

### `COMSOL_调试方法论.md`
- 只收录**已经被验证在不止一种场景下有效**的策略——如果只在一个项目里用过一次，先放
  进那个项目的`项目_*.md`里，等确认它在第二个不同场景下也有效之后，再"提炼"搬到这份
  方法论文件里。这样能避免这份文件被"其实是个例"的内容稀释。
- 每一条策略尽量写成"遇到X现象 → 用Y方法诊断 → 看到Z结果说明是A原因/继续查B"这种可
  直接照着执行的结构，不要只是叙述"我当时是怎么发现的"。

### `项目_*.md`
- 允许（且应该）比API手册/方法论文件更啰嗦、更叙事化——这里的目标是让下次接手这个项目
  的人（或AI）快速建立起"现在是什么状态、为什么是这个状态"的完整图景，而不是极简的
  速查表。
- 但**历史上被推翻/被更新公式覆盖的旧结论，应该整理成"最终结论+简要原因"，不需要保留
  完整的"试了A方案失败→试了B方案也失败→最后用C方案成功"这种逐帧过程**，除非那个失败
  过程本身包含值得记住的教训（比如"为什么直觉上合理的A方案会失败"）。
- 文件顶部应该有一个"当前参数/当前状态"速览区，以及"当前已知的开放问题"清单——这是
  下次会话最需要先看到的两块内容，其余按主题（不一定按时间顺序）组织。

## 何时应该新建文件 vs 加进现有文件

- 新项目（新的物理器件组合、新的目标应用，比如以后如果做磁扇形场质谱仪的完整项目而不
  只是组件测试）→ 新建`项目_<名称>.md`。
- 对现有项目的延续性工作（同一个模型改参数、修bug、加功能）→ 加进对应的`项目_*.md`，
  按主题归入合适的章节，不需要每次追加新章节到文件末尾。
- 发现一个新的、可复用的COMSOL组件/子系统（比如新的一种碰撞类型、新的一种物理场）→
  在`API手册.md`新开一个`§7.x`小节 + 更新§7.0表格。
- 发现一个新的、跨项目通用的排查策略 → 加进`调试方法论.md`对应主题的章节（不是每次都
  新建章节，先看现有的1-6节能不能放进去）。

## 本目录下的模型文件（.mph）与脚本

- 每个"项目"通常只保留**当前有效、最终版本**的建模脚本和对应.mph模型；被淘汰/被取代
  的中间版本脚本和模型应当删除，不要靠"注释里写着已废弃"来标记——具体见对应项目文件里
  "参考脚本"一节列出的当前有效脚本名单，那才是权威列表。
- 临时性的扫描/验证脚本（如`scan_d1.m`一次性用于寻找某个参数的最优值）如果其扫描结果
  已经被写进对应项目文档，脚本本身可以删除，不需要永久保留在目录里"以防万一"——文档里
  的表格/结论就是这次扫描的存档。

### 文件名前缀 = 生命周期约定（硬性规则）

| 前缀 | 生命周期 | 规则 |
|---|---|---|
| `scan_*.m` / `tmp_*.m` | **一次性**，用于寻找某个参数的最优值/做一次性对比 | 结果一旦写进对应`项目_*.md`的文档表格/结论，脚本本身和它产生的`.mph`/`.png`/`.log`**必须删除**，不允许"以防万一"长期保留。写这类脚本时脑子里就应该清楚它注定要被删。 |
| `test_*.m` | **永久保留**，独立组件的验证脚本 | 对应`COMSOL_MATLAB_API手册.md`§7.0部件库表格里的条目，任何项目都可能复用/参考，不要删。 |
| `ms_*.m` / `phase*.m` | **项目正式生产脚本**，长期维护 | 对应某个`项目_*.md`里"参考脚本"列出的当前有效版本。旧版本被新版本取代后才删除旧版本，正在用的不要动。 |

新建脚本时先决定它属于哪一类、按对应前缀命名，而不是事后才纠结"这个该不该删"。
