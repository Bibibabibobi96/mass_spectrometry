# 跨语言开发标准

本文是仓库内人类开发者和编码 Agent 共同遵守的代码实现与审查规范，适用于 Python、MATLAB/COMSOL
LiveLink、SIMION Lua/GEM、PowerShell及跨软件数据合同。仓库架构、参数权威、产物生命周期、GUI/CAD
门禁和Git规则仍只由根[`README.md`](../README.md)定义；Agent权限和交接行为只由
[`AGENTS.md`](../AGENTS.md)定义；项目物理状态只由项目`docs/PROJECT.md`定义。

本文使用“必须/禁止”表示硬要求，“应/优先”表示除非有明确技术理由否则采用的目标标准。标准没有
“旧代码永久豁免”：当任务触及不符合本规范的实现边界时，应在同一任务中完成相称迁移和回归；只有
迁移会实质扩大当前授权范围或无法建立可信回归时，才保留原实现并在项目权威文档记录有边界的后续项。
不得仅以改造麻烦、文件较长或调用者较多作为拒绝更好设计的理由。无关且稳定的文件不为追求形式批量
改写，避免把大范围格式变化与功能变更混在一起。

新增机器门禁若会一次暴露大量稳定旧代码，应采用可审阅的棘轮基线，而不是文件时间或Git提交日期：
基线逐项记录规则ID、路径、代码指纹、原因和退出条件；未变化的既有项可暂时保留，新违规必须失败，
旧文件修改到相关职责后原指纹失效并按新规则修复。复制、重命名或扩大豁免不得继承旧资格。安全、
物理正确性、数据损坏、权限和失败证据完整性问题不得进入兼容基线。

## 1. 设计与依赖边界

- 修改前搜索已有符号、入口、配置、Schema、测试和相邻实现；同一职责只有一个权威实现。
- 优先扩展现有接口。只有当前边界确实混合职责、阻止验证或不能表达需求时才新增抽象。
- 物理参数、单位、公式、默认值和指标定义不得在Python、MATLAB、Lua、PowerShell或GUI中平行维护。
- 求解器无关的理论、坐标变换、FWHM、TOF、传输率和统计分析不得依赖供应商API。
- 求解器实现负责准备供应商输入、执行、收集原始输出和转换到项目合同；一个求解器实现不得调用另一
  求解器的内部实现来伪造独立性。
- PowerShell只负责Windows环境、进程编排、路径、run生命周期和门禁，不实现物理公式或统计口径。
- 分析代码不应长期直接消费供应商私有格式；应由明确的转换边界生成项目canonical合同。一次性排错
  读取私有格式时，脚本名称、输出和文档必须标明诊断范围。
- 当前项目目录结构保持项目中心组织。未来统一求解器包装器按Roadmap实施；不得提前创建空的
  `domain/application/adapters`目录，也不得把未来接口写成当前已具备能力。
- 跨项目读取源码或配置时必须由消费项目的版本化依赖合同显式声明提供项目、仓库相对路径和冻结
  `run input`身份；运行器不得目录搜索“可用文件”，必须把依赖复制进本次`inputs/`，核对源与副本
  SHA-256，并在run config和manifest中保留身份。稳定公共API形成后可改为API依赖，但仍须冻结版本。

依赖方向应保持为：机器合同和纯领域逻辑不依赖求解器；编排层调用领域逻辑、求解器实现和I/O；
供应商脚本只消费已冻结的输入。若出现循环导入、多个位置散布`if solver == ...`或为运行而反复修改
模块搜索路径，应重新设计边界，而不是增加补丁分支。

## 2. 命名

同一概念跨语言使用同一英文术语。已有机器合同字段是该概念的命名权威，不能为了偏好创建同义字段。

|对象|标准|示例|
|---|---|---|
|Python、Lua、MATLAB文件|`snake_case`|`particle_state.py`|
|Python函数和变量|`snake_case`|`compute_time_of_flight`|
|Python类、dataclass、异常|`PascalCase`|`RunContractError`|
|常量|`UPPER_SNAKE_CASE`|`DEFAULT_TIMEOUT_S`|
|JSON键|`snake_case`并显式单位|`position_x_mm`|
|CLI子命令和选项|`kebab-case`|`--run-id`|
|测试|`test_<行为>_<条件>`|`test_fwhm_rejects_empty_input`|
|PowerShell公开函数|批准的`Verb-Noun`|`Invoke-SolverRun`|

布尔量使用肯定且无歧义的名称，优先采用`*_enabled`、`*_allowed`、`*_required`、`*_passed`、
`is_*`或`has_*`。禁止`flag`、`bool1`等无法独立解释的名称。既有合同中的`physical_link`等明确字段
无需机械改名。

动词语义固定如下：

- `load_*`从外部存储读取，`parse_*`解释表示，`validate_*`执行失败关闭的合同校验；
- `prepare_*`准备但不启动昂贵工作，`run_*`或`execute_*`真正执行；
- `compute_*`执行无副作用数值计算，`convert_*`转换单位或格式，`normalize_*`生成项目统一合同；
- `write_*`或`export_*`持久化，`compare_*`产生显式比较结果，`ensure_*`保证后置条件；
- `get_*`只用于低成本获取已有值，禁止隐藏求解器运行、网络请求或大文件读取。

文件和函数名描述单一职责。禁止新建无明确边界的`utils.py`、`helpers.py`、`misc.py`、`data2.py`、
`final_analysis.py`。根`common/`是README定义的共享边界，不属于模糊命名。源码和运行文件不得用
`final/latest/new/v2/retry`管理版本；版本、状态和重试身份使用Git、Schema及README规定的容器命名。

## 3. 单位、坐标与机器合同

- 系统边界的数值字段必须在字段名或Schema中显式单位，不得隐式混用mm/m、µs/s、eV/J或Da/kg。
- 不强制把所有内部量转换为SI。选择能减少转换和数值错误的项目canonical单位，并在边界只转换一次。
- 坐标合同必须给出frame身份、轴方向、原点、旋转和平移；位置应用旋转和平移，速度只应用旋转，
  时间和能量不得因空间变换静默改变。
- 新跨模块机器合同必须定义`schema_version`、字段、类型、单位、空值、范围、枚举和兼容策略，并提供
  JSON Schema或等价的版本化校验器。现有合同在被实质修改时补齐，不因格式偏好重写稳定物理真值。
- 缺失字段、未知的物理关键字段、非法范围、重复身份、非有限数值、单位或frame冲突必须在昂贵运行前
  失败。是否允许扩展字段由Schema显式决定，不能由解析器静默忽略。
- 环境变量可以把已由冻结合同解析的路径和数值传给MATLAB、COMSOL或SIMION，但不能成为物理参数、
  默认值或实验设计的唯一来源，也不能反写baseline。
- canonical粒子状态必须保持身份、物种、三维位置、三维速度及适用的全局时间语义。不得另建字段名称
  不同但数值相同的第二套状态表。

## 4. 函数、模块与复杂度

行数是审查信号，不是追求形式的机械目标。拆分依据是职责、可测试性和依赖边界。

### Python

- 普通纯计算、转换或校验函数以60逻辑行以内为目标；超过100行时必须审查是否混合多个职责。
- 普通模块超过约600逻辑行时必须审查是否应按职责拆分。理论推导、静态表和清晰的单一流水线可例外。
- 嵌套通常不超过3层，优先使用guard clause、具名策略和小型纯函数。
- 多个参数共同描述一个请求或状态时，优先使用dataclass、TypedDict或明确的请求对象；不要用多个布尔
  参数切换互不相关的模式。
- 公共函数必须有类型注解。复杂返回值使用具名结构，不返回含义不明的裸tuple。
- 禁止可变默认参数和无边界的可变全局状态。

### MATLAB/COMSOL、Lua与PowerShell

- 不设统一行数硬上限。模型树构建和失效安全run编排可以较长，但几何、物理、网格、Study/Solver、
  跟踪、导出和manifest阶段必须能被清楚识别并独立校验。
- 重复逻辑、独立可测试算法或混合的物理/编排职责必须拆分；不要为了缩短文件创建大量只转发参数的
  单调用者函数。
- MATLAB主函数名必须与文件名一致，禁止依赖base workspace、隐式`cd`、`eval`或未保存变量。
- 只构建几何的MATLAB任务必须在文件头声明`% REPOSITORY_CONTRACT: MATLAB_BUILD_ONLY`；声明后禁止
  创建Mesh、Physics、Study或Solver，禁止提取解、运行粒子及保存MPH。几何`geom.run`和几何信息检查
  仍允许，由轻量开发标准门禁失败关闭。
- Lua变量默认`local`，事件回调保持短小；禁止在逐粒子回调中反复打开文件。
- PowerShell使用`Set-StrictMode -Version Latest`和`$ErrorActionPreference = 'Stop'`；物理和统计逻辑
  应迁到可测试的Python或项目理论实现。

圈复杂度暂不设统一数值门禁。若一个函数无法用小型、可枚举的测试矩阵覆盖其分支，或修改一个条件会
影响多个无关阶段，就必须拆分。未来选择复杂度工具后再以仓库实测分布确定数值阈值。

## 5. 注释、公式和文档字符串

- 注释解释约束来源、设计理由和风险，不逐行翻译代码。
- 复杂物理公式必须说明符号、单位、适用域、近似条件和权威理论来源，并只实现一次。
- 新增或实质修改的公共Python模块、类和函数应提供简洁docstring，说明输入、输出、单位、异常和副作用。
- TODO必须包含具体动作、适用条件或责任上下文；禁止`TODO: fix later`。
- 禁止大段注释掉的旧实现和无调用死代码；Git保存历史。
- 供应商生成代码、静态表和人工维护包装器必须能从目录、文件头或邻近README区分。

## 6. 外部进程、失败与日志

- Python调用外部进程时使用参数数组，避免`shell=True`和拼接命令；确需shell语义时必须限制输入并解释原因。
- 显式设置工作目录、超时、编码、日志路径和必要环境；不得依赖调用者当前目录、系统locale或目录遍历顺序。
- 记录命令时移除token、许可证内容和敏感环境值。不得把个人绝对路径写入Git内合同；运行证据中的主机
  路径由manifest管理。
- 成功不能只凭文件存在或本地化GUI文本判断，必须结合进程/API终态、必需输出和结果合同。
- 超时先请求正常终止，再按公共入口策略终止本次进程树；不得误杀任务开始前已存在的共享进程。
- 自动重试只允许幂等瞬态故障，必须保持同一冻结输入并记录次数和最终原因。配置错误、合同失败、模型
  错误和数值发散禁止自动重试；商业工具的具体白名单以相邻公共入口和软件文档为准。
- 禁止空`except`、打印后继续或返回空数据冒充成功。转换异常类型时保留原异常链。
- 公共生命周期代码应区分配置、准备、许可证、启动、超时、原始输出和结果合同错误；小型脚本可以使用
  清楚的内建异常，不为形式建立无用异常层次。
- 原生供应商日志原样保存。新公共包装器的结构化索引目标字段为`timestamp、level、run_id、project、
  mode、solver、stage、event、message`，但不得为统一格式丢失原始日志。

## 7. 性能、缓存与数据量

- 优化前建立正确性baseline和分阶段耗时；没有测量不得声称加速。
- 性能改动前后必须保持相同输入、种子、精度、网格、时间步、粒子数、终止条件、保存字段和指标定义，
  或把每项有意变化作为独立变量和物理结论重新验证。
- 未变化的场、模型或解析发布可以复用，但必须由配置、代码、模型、网格/场、求解器版本和关键选项
  的内容身份证明安全。文件名和修改时间不足以作为缓存键。
- 缓存是可删除的性能层，不能代替不可变run证据；缓存命中必须验证元数据，损坏或不完整时重新生成。
- 批量处理粒子，禁止为每个粒子启动外部进程、解释器或跨语言会话。
- 默认只保存分析所需字段；密集轨迹只在接口误差、碰撞、空间电荷、脉冲捕获或明确诊断需要时保存。
- 大表按块读取和聚合。CSV继续用于可审阅的小中型合同；只有测量证明CSV成为主要瓶颈并建立版本化
  Schema后，才引入Parquet或HDF5。
- 并发受CPU、内存、I/O、许可证数和供应商条款限制；禁止嵌套并行造成超额占用。
- 默认数值精度为`float64`；降低精度必须提供误差分析。O(N²)及更差算法必须说明预期N上限和扩展方案。
- 性能测试与正确性测试分离，报告机器、软件版本、重复次数和统计量；共享机器优先使用中位数。
- 新生命周期实现优先在临时或scratch路径完整写入、校验，再原子发布到目标；任何原子替换仍受README
  生命周期和AGENTS删除授权约束。

## 8. 测试

- 纯函数、单位、坐标、合同和指标使用无需商业许可证的快速测试。
- 测试同时覆盖有效与无效边界，包括空结果、NaN/Inf、重复ID、缺列、非法时间、未知状态和无到达。
- 外部入口按适用范围覆盖异常退出、超时、许可证失败、缺失输出和残留文件误判。
- FWHM、质心、传输率、分辨率及其他共享指标必须有冻结定义和独立参考向量。
- 修复可复现缺陷时增加最小回归测试；不得只修改当前样本使其通过。
- 浮点容差必须来自解析误差、数值收敛、序列化精度或物理预算。禁止任意放宽容差、删除异常粒子、
  修改baseline或挑选最好种子来制造PASS。
- 随机测试固定种子；统计结论使用稳定区间、配对设计或多种子验证，不依赖偶然单值。
- 纯单元测试使用小型fixture，不依赖偶然artifacts。Candidate、Formal和历史缺陷复验可以显式消费已
  验证manifest，但必须校验哈希和来源身份。
- 商业许可证、大模型和GUI/CAD测试必须与轻量门禁分层；跳过时明确说明结论边界。

不强制把项目测试迁移成`unit/integration/regression/performance`四棵目录。现有`analysis/comsol/simion/
cross_solver`结构更直接表达工具和证据边界；测试性质由入口、门禁层级和文档说明。

## 9. 安全与完成条件

- 不执行未经审查的外部脚本、宏、模型或二进制；用户提供不等于已确认其行为安全。
- 不在任务中自动下载或升级求解器、插件、运行时或依赖，除非用户明确授权且版本被记录。
- 禁止伪造求解器输出、日志、性能数据和测试结果，禁止通过改变输入或精度隐藏失败。
- 完成代码变更时按适用范围运行格式/语法、静态门禁、单元测试、集成测试和真实求解器验证；没有统一
  工具的语言不为勾选清单临时引入格式器。
- 可机器判定的硬要求由`common/verify_development_standards.py`检查并接入轻量门禁；`--show-review`
  可列出函数和模块长度审查信号。长度信号只要求人工审查职责，不自动判为违规。
- 交接必须区分代码检查、求解器无关分析、真实求解器、GUI/CAD和Formal层级，报告未执行且影响结论
  的验证及剩余风险。常规命令和日志仍按AGENTS的紧凑测试报告规范汇总，不逐条复述。

本规范不要求每次变更验证所有求解器、建立性能报告或测试无关错误路径；要求的是验证范围与实际改动、
风险和声明相称。声称性能改善时必须有性能数据，改变外部进程或失败处理时必须测试相关异常路径，改变
物理或机器合同时必须同步合同、实现、测试和最近的权威文档。
