# 质谱生成式工程平台路线图

本文将[`VISION.md`](VISION.md)的长期目标拆成有先后依赖的能力阶段。它是跨项目方向和项目族规划的
唯一权威，不替代各项目`docs/PROJECT.md`中的当前状态、开放问题和短期下一步，也不把目标示例提升为
参数基线。阶段以能力门禁而非日历日期结束。

## 规划原则

- 先建立可审阅的规格和可靠闭环，再增加优化规模和自然语言自主程度。
- 先在一个具体项目验证能力，再判断是否提升为跨项目工具。
- 一个项目对应一条可独立正式化的硬件设计线；设计族通过元数据关联，不用深层目录包住多个formal。
- 同一硬件仅改变电压、频率、气体、粒子源或运行目的时使用mode；参数扫描和优化候选使用run。
- 电极拓扑、主要功能、正式资产或验收合同需要独立长期维护时，建立新的平级项目。
- 每一阶段都必须保留失败、不可行性和适用边界，不能只保存最优候选。

## 面向大规模代码库的扩展护栏

仓库增长到十万行乃至更大规模时，主要风险不是源码行数本身，而是求解器入口、物理合同、派生参数、
项目间依赖和运行记录形成多份不一致真值。以下任务按实施成本分层；它们不改变当前项目阶段，也不得
打断已批准的器件功能闭环。

获得批准后可随近期改动渐进实施的低成本护栏：

- 新增或正在修改的求解器代码分离几何、网格、物理、求解、跟踪和导出职责；build-only入口不得通过
  条件分支隐式进入求解，旧大型脚本只在被实际修改时分段收缩，不做无目标的全仓重写；
- 几何、位姿、时间、粒子和数值参数只从版本化合同派生；静态测试拒绝在第二个入口重复关键物理硬
  数字，并核对阶段状态、权限和证据run的一致性；
- 在单个项目内收敛PowerShell运行器重复的三件套初始化、环境恢复、失败收尾和manifest写入模式；
  只有两个项目实际复用并通过各自门禁后，才把稳定辅助层提升到`common/`；
- 为跨项目取用源码增加显式依赖身份和静态边界检查；运行包可以冻结依赖副本，但来源路径、版本和
  SHA-256必须进入manifest，禁止从相邻项目任意搜索“可用脚本”；
- 当前状态文档只保留现状、权威证据、限制和开放任务；关闭过程及时冻结到扁平history，避免项目入口
  随运行次数线性增长。

需要独立设计、迁移和回归预算的未来平台任务：

- 建立统一求解器执行合同和适配器入口，替代各项目重复的COMSOL MATLAB、SIMION Lua和其他求解器
  编排代码，同时保留软件专用模型逻辑；
- 为稳定JSON合同建立Schema注册、版本迁移和类型化加载/派生层，再逐步迁移现有合同；不能一次批量
  改写全部配置并假定语义等价；
- 建立源、连接器、离子光学部件、分析器和检测器的组件API及单向依赖图，统一坐标、全局时钟、粒子
  身份和稀疏状态语义；
- 建立依赖感知CI，在保留定期全仓门禁的同时，只对普通提交运行受影响的契约、项目和集成测试；
- 从现有manifest生成可重建的轻量artifact索引，避免随大型MPH和run数量增长而每次递归扫描、重复
  哈希全部载荷；索引不是第二份结果真值，损坏时必须能从manifest重建；
- 对现有单体脚本和旧运行器的全量模块化属于专项迁移，必须按项目逐个保持GUI、数值和artifact回归，
  不与正在推进的物理功能阶段混合提交。

启动未来平台任务的触发条件包括：相同运行协议已在至少两个项目稳定复用、一次合同修改需要同步三个
以上入口、轻量门禁时间显著妨碍普通提交，或artifact清单扫描成为日常等待的主要部分。完成判据不是
“增加框架”，而是减少重复真值、缩小变更影响范围，并保持现有正式资产和失败证据可追溯。

## 设计族与未来项目

设计族用于自然语言需求分类、模板选择和共享方法发现，不是artifact目录层级，也不拥有含义不明的
单一`formal/`。

### TOF质量分析器族

- 当前`oa_tof`继续作为现有正交加速、单次折返设计线；双级环栈反射器表示两个场区，不等于两次反射。
- 环厚、环宽、间距、孔径、屏蔽、漂移长度和电压的优化属于当前设计的候选run。
- 飞行拓扑变为两次独立折返时，建立`oa_tof_double_reflection`等新的平级项目。
- MR-TOF具有多次往返和独立验收体系，已建立独立`mr_tof`原型项目接收用户自绘CAD种子；后续理论、
  参数化和验证在该项目内逐步建立，不放进当前`oa_tof/formal`。
- 若多个TOF项目实际复用同一分析或生成器，再将经验证的部分提升到`common/tof/`。

### RF多极杆离子光学族

- 当前`rf_quadrupole_collision_cooling`保持原项目名和artifact身份，不改成宽泛的多极杆容器。
- 已建立平级的`rf_hexapole_ion_guide`和`rf_octupole_ion_guide`，分别维护六极杆、八极杆离子导引的
  独立合同和证据；未来可继续建立`rf_quadrupole_mass_filter`、`rf_hexapole_collision_cell`等设计线。
- 同一正式硬件的无碰撞传输、碰撞冷却、气体或RF工况可作为mode；不同杆数、主要电极拓扑或独立
  质量响应验收通常形成新项目。
- 六极杆和八极杆已共同验证的理想解析场、L1有限长度传输、二维圆杆L2和封闭接地腔有限三维功能
  进入`common/multipole/`；开孔接口、网格收敛、机械资产和通用求解器适配尚未形成双项目证据。

### 离子源、电子枪与接口族

- `electron_impact_ion_source`和`wehnelt_electron_gun`先补齐机器可读参数链与独立部件验收。
- 新离子源或电子枪只有在拓扑和正式资产可独立维护时才建立项目；仅工作点变化保留为mode。
- 部件集成不以文件拼接为完成条件，必须冻结坐标、电位、孔径、压力、相空间、时间和机械接口合同。

## 阶段一：需求与项目注册

目标是把自然语言目标变成可审阅而非直接执行的结构化规格。

主要能力：

- 建立项目注册信息，显式记录`project_id`、`family_id`、结构类型、主要功能、正式能力和适用接口；
- 定义需求字段、单位、硬约束、优化目标、优先级、容差、证据等级和未决问题；
- 将自然语言解析结果冻结为候选规格，要求人工或规则门禁批准后才能创建求解run；
- 区分目标示例、已批准目标、当前baseline和正式验收结果。

完成门禁：至少OA-TOF和当前RF四极杆能够从同一规格框架产生可追溯run config，且含糊或冲突需求
会在求解前被拒绝。

## 阶段二：OA-TOF性能驱动闭环

OA-TOF作为首个端到端性能设计示范。输入可以是质量、电荷态、目标分辨率和尺寸/电压限制，也可以是
部分冻结几何；系统自动建立候选、运行多保真度模拟并依据诊断调整参数。

设计空间逐步覆盖：加速区、漂移区、反射器和检测器位置；环电极数量、厚度、宽度、间距和电压；
孔径、端盖、屏蔽壳和开口；局部场分量、边缘场、飞行时间聚焦、传输率和落点分布。轴对称圆形
加速器候选属于这一阶段的具体结构分支。

类似“500 Da、单电荷、分辨率30000”只在批准后成为目标规格；优化器必须同时报告传输、鲁棒性、
数值收敛和制造约束，不能只优化单一R值。

完成门禁：从批准规格到正式COMSOL、SIMION、CAD、结果和manifest的闭环至少完成一次；未达标时
能够输出可复验的瓶颈与不可行性证据。

## 阶段三：多极杆功能设计

在保持当前四极杆回归和接口证据的基础上，按独立项目逐步扩展功能与杆数。六极杆和八极杆已经完成
理想L1、二维圆杆L2和封闭接地腔有限三维功能基线；它们仍不代表完整接口、机械设计或正式资产完成。

建议顺序：

1. 六极杆、八极杆离子导引：从已完成的封闭接地腔有限三维功能推进到参数化开孔接口、外部注入、
   输出相空间和下游匹配；
2. 四极杆质量过滤：质量扫描、峰位、分辨率、透过率、稳定区和入口/出口边缘敏感度；
3. 四极杆或六极杆碰撞单元：传输、冷却、碎裂、能量和碰撞统计；
4. 仅将至少两个项目已验证的理想场、几何生成或结果能力逐项提升到公共层。

完成门禁：自然语言功能要求能够选择合适的杆数与项目模板，完成至少两种不同功能或杆数的独立正式
设计，并证明共享工具没有把某个项目的假设伪装成通用规则。

## 阶段四：离子源与电子枪自动化

为电子轰击离子源和Wehnelt电子枪建立`baseline → resolved → solver/CAD`链，定义发射、束流、能量、
空间电荷、热、磁场、离化效率和机械约束等适用指标。先完成独立部件验证，再允许它们向下游输出
冻结接口状态。

完成门禁：至少一个源项目能从批准规格生成正式资产和接口分布；另一项目实际复用相同规格或验证
基础设施后，才提升跨项目源设计能力。

## 阶段五：部件集成与仪器级优化

建立源、多极杆、TOF和检测器之间的接口合同与组合门禁。两端独立相空间可以送入同一冻结下游模型；
不要求随机碰撞后的逐粒子轨迹重合，但必须验证接口接受率和最终功能。

优化从单部件扩展为系统目标：总体尺寸、电压和压力预算，传输、分辨率、灵敏度、鲁棒性、制造与计算
成本。部件不能为了改善局部指标而破坏上下游接口。

完成门禁：至少两类正式部件在冻结接口下组成可复验系统，系统级正式结果能够追溯到全部来源项目、
版本、变换和运行manifest。

跨项目网格建设与计算量控制统一遵循[`VALIDATION_METHODS.md`](VALIDATION_METHODS.md#网格建设与计算量控制)。
本阶段的未来平台任务是把该方法实现为公共网格合同、求解器适配器和结果manifest字段；当前各项目
先保存可复验的局部证据，不在本轮RF→oa连接任务中提前抽取未经多项目验证的通用代码。

## 阶段六：受控自然语言自治

在前述规格、项目注册、优化、验证和正式化能力均稳定后，形成面向使用者的自然语言工程入口。系统
能够提出必要澄清、解释设计选择、管理长时间运行、根据失败证据调整候选，并在达到门禁后准备正式
发布；涉及目标变更、不可逆资产晋升和重大范围扩张时仍需明确批准。

- 未来建立统一求解器包装器（例如`tools/run_solver.py`），由一个版本化JSON执行合同和统一CLI分派到
  COMSOL、SIMION及以后实际引入的GPT等适配器；调用方只提交项目、solver、config、粒子数和run id。
  适配器保留各软件专用MATLAB/Lua/输入文件与验证逻辑，但统一产生`inputs/`、`logs/`、可选`raw/`、
  `results/`以及根`run_config.json/summary.json/run_manifest.json`。这是未来平台任务，不是当前已具备
  的单一运行入口；在各项目包装器和输出语义稳定前不提前抽象实现。

完成门禁：多个不同部件族能够从自然语言需求走完同一受控生命周期；重复运行可复现，失败不会被
隐藏，正式资产不会绕过参数、GUI、CAD和跨求解器门禁。

## 暂不提前固化的内容

- 不在实现模块尚未形成前编写推测性的完整`ARCHITECTURE.md`；实际出现规格编译器、设计注册、优化器、
  求解器适配和发布器后，再记录稳定接口与依赖方向。
- 不为未来项目预建空目录、空formal或无验证的通用代码。
- 不给阶段承诺日期；具体优先级和资源变化时更新本路线图，各项目当期工作只更新其`PROJECT.md`。
- 不把双反射OA-TOF、MR-TOF、多极杆新功能或整机集成写成当前已具备能力。
