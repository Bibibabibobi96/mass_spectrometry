# 项目执行规则（跨 Agent 共用）

本文件是 Claude Code、Codex 及其他编码 Agent 共用的执行约束。开始任务先完整阅读
`README.md`；它规定阅读顺序、知识写入位置、目录职责、生命周期、GUI 对等、CAD 同步和 Git
规则。随后只按目标项目 README 的路由读取 PROJECT 与实际使用的软件文档，不要默认通读历史或
所有软件文档。

## 必须遵守

- 开始修改前先搜索已有实现、机器合同、测试和相邻入口；优先扩展现有边界，不创建同义函数、同义字段、
  第二CLI或只为假想需求服务的抽象。不得只凭文件名推断模型、二进制、数据或历史证据的内容。
- 修改正式项目脚本前，先阅读项目 `README.md` 和 `docs/PROJECT.md`；旧 `项目_*.md` 只在尚未完成新结构迁移的项目中暂作入口。
- 修改或审查代码时完整阅读[`docs/DEVELOPMENT_STANDARDS.md`](docs/DEVELOPMENT_STANDARDS.md)；该文档
  约束人类和Agent共同遵守的编码、合同实现、进程、性能与测试标准，不改变本文件的Agent权限或
  根README的仓库架构和产物生命周期。
- 脚本分类、产物生命周期和保留条件以根README对应章节为唯一权威；Agent只额外执行本文件规定的
  删除授权、任务内归属和交接责任，不在这里维护第二份分类表。
- 删除任何文件，尤其`.mph`模型和计算结果前，必须先取得用户确认。
- 修改项目脚本时同步更新最近的项目文档；新增跨项目 COMSOL API 经验更新`docs/COMSOL_API.md`，仅跨项目验证过的策略才更新根 `docs/`。
- 新增或修改规范前，先按根README的权威表确定唯一写入位置，并用全文搜索检查是否已有同义规则；
  非权威入口只链接或简述适用范围，不复制规范正文。
- 不建立 COMSOL、SIMION、CAD 文档之间的横向网状引用；项目 README 负责导航，统一结论写入 `docs/PROJECT.md`。
- MATLAB、COMSOL、Python和SolidWorks的版本及启动方式只以根README的
  [“工具链与执行入口”](README.md#工具链与执行入口)为准；Agent不得另建连接生命周期或把临时检查工具当作正式入口。
- PowerShell Core 7（`pwsh`）是仓库唯一受支持的PowerShell运行时；公开根入口必须通过公共preflight
  fail-closed验证当前宿主为Core 7，禁止回退到Windows PowerShell 5.1。Python等非PowerShell边界启动
  PowerShell时必须使用`pwsh`，PowerShell内部脚本继承当前宿主，不得另行选择或启动第二套PowerShell。
- MATLAB/COMSOL和SolidWorks会写用户配置、许可证、日志及临时目录；Agent调用这些商业工具时必须从
  第一次尝试就使用允许其正常访问用户配置目录的执行上下文。公共入口报告
  `EXECUTION_ENVIRONMENT_BLOCKED`时，应在权限正确的上下文重试同一冻结输入，不得把它归因于模型、
  求解器或物理FAIL，也不得靠延长求解超时绕过。
- GUI对等与CAD同步的技术判据只由根README的“GUI 与 CAD 门禁”定义；Agent必须按变更类型执行该
  门禁，未完成适用验收时不得报告正式模型或正式几何已经完成。
- 禁止伪造求解器输出、日志、性能数据或测试结果，也禁止用空表、残留文件、降低精度、放宽无依据容差、
  删除异常粒子或改变输入来制造PASS。外部脚本、宏、模型和二进制在执行前必须确认来源与任务内用途。
- 昂贵求解或批量生产运行前，先完成与风险相称的合同校验、静态检查、纯分析测试或最小工况；环境预检
和小样本不得冒充真实求解器、统计样本或Formal验证。修复可复现缺陷时应加入能阻止其复发的最小测试。
- 各项目的默认首要目标是正确贯通几何、物理、场、粒子状态、时间、数据和证据链，并确保全程可追溯。
  除非用户明确指定优化目标、指标和约束，不得把任务自主扩展到数据优化、性能调参、N=1000统计、
  参数扫描、S4或其他后续阶段；功能PASS只证明声明范围内的功能贯通，不得表述为优化、性能最优或
  Candidate/Formal资格结论。

## 多 Agent 协作与自主委派

- 主 Agent 可以根据任务结构自主调用 sub-agent，不必逐次取得用户许可。自主调用表示获得委派权限，
  不表示必须使用尽可能多的 Agent；应使用能够显著提高效率或审查独立性的最少数量。
- 在合理且不会增加冲突、资源争用或整合成本的情况下，主 Agent 应把委派作为大多数可分解任务的
  默认工作方式，尽可能由 sub-agent 承担边界明确的资料审查、实现、测试和独立复核；主 Agent 负责
  整体规划、任务划分、用户交互、全局决策、冲突协调、跨工作流整合和最终报告。这里的“尽可能”
  指提高可安全委派工作的占比，不是增加无必要的 Agent 数量；存在合适独立任务和空闲执行槽位时
  应优先委派，不得因主 Agent 亲自执行可安全委派的长任务而长期延迟响应用户。
- 主 Agent 在面向用户的进度报告和 updated plan 中编号为 `Agent 0`；同一任务内的 sub-agent 按
  `Agent 1`、`Agent 2`……使用稳定编号，并同时报告各自任务和状态。除非 Agent 被关闭后承担了全新
  工作流，不得在同一任务中更换编号或只用易混淆的内部名称代替编号。
- 满足以下任一条件时，优先考虑委派：
  - 任务能够拆成两个或以上边界清楚、可独立推进的工作流；
  - 存在耗时较长但不依赖主流程继续推理的检查、测试或资料审查；
  - 需要代码、文档、数据链或物理结论之间的独立复核；
  - 不同目录或项目可以由不同 Agent 分别负责，且并行工作能明显缩短总耗时。
- 以下情况通常不调用 sub-agent：
  - 单文件、小范围或短时间即可完成的任务；
  - 后续步骤严格依赖前一步结果，无法形成有效并行；
  - 多个 Agent 必须频繁修改同一文件或同一紧耦合代码区域；
  - 并行运行会争用同一个 GUI、商业软件实例、许可证、Git 索引或高负载计算资源。
- 委派前必须明确：
  - 具体目标和完成判据；
  - 允许读取和修改的目录或文件范围；
  - 禁止修改的相邻范围；
  - 必须执行的验证；
  - 交付内容以及是否由该 sub-agent 创建独立提交。
  不得把“继续检查”“处理剩余问题”一类无边界任务直接交给 sub-agent。
- 同一文件在同一时段只能有一个写入责任 Agent。只读审查可以并行，但审查 Agent 未取得明确写入
  责任前不得修改文件。涉及多个工作流的共享文件、跨目录重构和冲突整合由主 Agent 统一处理，或在
  重新划定唯一写入责任后交给一个 Agent。
- 每个 sub-agent 开始工作时仍须完整读取根 `AGENTS.md`，并按根 README 的固定顺序读取其任务实际
  需要的项目文档和规范；主 Agent 的任务摘要不能替代仓库权威文档。
- COMSOL、SIMION、MATLAB、SolidWorks以及其他会占用独占进程、许可证或大量内存的操作默认串行。
  主 Agent 应维护明确的运行次序；只有已有证据证明并行安全且不会混淆产物归属时才可并行。
- Git 暂存、提交和推送操作必须串行。sub-agent只能提交其明确负责且已经完成相称验证的主题，不得
  暂存或提交其他 Agent 的改动；开始 Git 写操作前应通知主 Agent，避免共享索引竞争。提交与推送
  仍须遵守本文件的“Agent 自主提交”和根 README 的 Git 规则。
- sub-agent发现会改变任务方向、影响结果可信度、造成文件范围重叠或阻塞后续工作的实质问题时，
  应立即报告主 Agent；重要问题解决和任务完成时也应立即报告。主 Agent应及时向用户转达这些节点，
  不等待全部并行任务结束后再集中报告。常规命令、无影响的中间日志和重复进度不逐条报告。
- 主 Agent始终负责整体一致性：检查各工作流是否冲突、整合结论、执行必要的联合门禁、确认提交边界，
  并向用户给出完整的最终交接。sub-agent完成局部任务不等于整个用户任务已经完成。
- 如果用户要求“先报告、批准后修改”，所有 Agent 在批准前只能进行只读审查；自主委派不得绕过用户
  明确设置的审批点、删除授权、外部写入限制或其他权限边界。

## 测试报告（面向用户）

- 完成一次或多次测试后，最终交接必须给出测试报告。常规单阶段任务通常控制在**1000 tokens以内**；
  这是篇幅指引而非硬性上限。任务包含多个独立阶段、执行时间较长并产生较多需要比较的结果、存在
  失败排查或数据完整性风险、需要说明复现边界，或用户明确要求详细报告时，可以合理超过。篇幅由
  有效结论和决策信息决定，不因常规日志或重复过程增加；长任务按阶段汇总自上次交接以来的新结果，
  不重复复述已归档过程。
- 按以下紧凑顺序报告：**目标与判据**（测什么、何时算成功）；**对象与变量**（模型/样本量、对照与唯一改动）；**结果**（关键数值用表格）；**结论**（是否接受假设、适用范围）；**产物**（脚本/模型是否改动、临时产物清理、项目文档与提交）。
- 不逐条复述常规启动命令或无关日志；初始化/运行失败仅在它影响结果可信度、改变测试路径或需要复现时简述。多轮测试合并到同一张结果表，明确每轮的样本量和关键差异。
- 明确区分源码/静态门禁、求解器无关分析、真实求解器运行、GUI/CAD复验和Formal资格；列出因任务范围、
  环境、许可证或用户决定而未执行且会影响结论边界的关键验证，不用“全部通过”概括未运行的层级。
- 测试产生的项目特有结论（包括明确的负结果）仍须按本文件及`README.md`的知识归档规则写入对应项目文档；面向用户的短报告不替代项目记录。

## Git 约束

- 主题拆分、明确暂存、提交正文和提交前检查遵守根README的“Git规则”；Agent不得用聊天记录
  代替复杂提交所需正文，也不得因文件未跟踪而忽略仓库根工具崩溃日志的归属与manifest责任。
- 每个提交都必须同时包含标题和有信息量的正文，正文概括目的、结果或行为变化、实际验证及必要限制；
  即使是机械性或单点修改也不得只写标题。
- 不覆盖、回退或夹带已有未提交改动；修改同一文件前先检查现有差异。

## Agent 自主提交

- Agent 在一个独立主题完成且已完成相称验证后，应自行创建提交，不必等待用户催促。典型时机是：一个功能/模型结论完成、一个文档规则整理完成、一个目录重构完成，或即将切换到无关主题之前。
- 若一次任务包含多个可独立审阅的主题，按主题分别提交；不要为追求更少提交而合并它们。
- 只提交本次任务产生或已明确纳入当前主题的改动。任务开始前已有的无关改动保持不动；若改动混在同一文件，先检查差异并仅在能安全拆分时提交。
- 不在结果未验证、变更范围不清楚、存在合并冲突、或用户明确要求暂不提交时自动提交。无法运行验证时，在提交说明或最终交接中写明原因。
- 本项目远程仓库是私有仓库。完成上述验证并创建提交后，Agent 应自动使用普通 `git push` 推送当前分支到其 `origin` 跟踪分支，不必逐次等待确认。
- 禁止 `--force`、`--force-with-lease`或任何远程历史改写。推送失败时保留本地提交并报告，不用替代手段绕过认证、分支保护或冲突。
- 按用户 2026-07-15 的明确决定，不增加敏感信息专项扫描；现有暂存范围、禁止二进制/结果文件和测试检查仍然有效。
