# 质谱仿真源仓库与文档使用指南（AI 与人类共用）

> 这份文档是"文档的文档"——告诉你（无论是AI还是人）这个目录下每份经验文档是干什么的、
> 遇到什么情况该查哪一份、新发现该往哪份文档里加、加的时候要遵守什么规则。**每次开始
> 新任务前，先看这份指南决定去读哪份文件，不要每次都从头翻旧的单体大文档。**

## 文档体系总览

本目录下的经验文档分成四类，边界很明确：

| 文件 | 回答什么问题 | 什么时候查/什么时候写 |
|---|---|---|
| `COMSOL_MATLAB_API手册.md` | "这个COMSOL/MATLAB调用该怎么写？这个属性叫什么名字？合法取值是什么？有没有现成验证过的部件可以直接复用？" | 写任何新脚本、建任何新几何/物理场/求解器配置之前先查§7.0部件库表；踩到"某个调用报错/行为诡异"时先查§7.12黑名单和对应小节。**任何项目**（不管是质谱仪哪个部件、电子枪、还是以后的新项目）都可能用到。 |
| `COMSOL_调试方法论.md` | "结果不对/比预期差，但不知道从哪查起，该用什么策略缩小排查范围？"、"服务端卡死/崩溃了怎么恢复？"、"这类几何/选择集问题有没有通用陷阱？" | 遇到"分辨率不达预期"、"场精度有偏差但不知道是谁的问题"、"MATLAB/COMSOL卡死"这类**方法论层面**的困境时查。这里的每一条都是跨项目验证过的策略，不是某个项目的具体参数。 |
| `components/mass_analyzers/oa_tof/dual_stage_ringstack/docs/项目_oaTOF双级环栈反射镜.md` | "这个具体的oa-TOF双级环栈反射镜分析器，现在的参数是什么？为什么是这些参数？踩过哪些只在这个几何上才会出现的坑？" | 修改该组件`comsol/ms_*.m`或任何`scan_*.m`脚本之前必须先读，了解当前状态和历史教训，避免重复踩坑或推翻已验证的结论。 |
| `components/electron_sources/wehnelt_electron_gun/项目_螺旋灯丝Wehnelt电子枪.md` | "螺旋灯丝+Wehnelt电子枪项目的物理结论是什么？" | 修改该组件`phase*.m`系列脚本之前查。 |

`official_docs/`目录存放COMSOL官方PDF手册，作为离线原始参考，不属于"经验文档"，一般不要全文阅读。
遇到本地经验手册没有覆盖的问题时，先看`official_docs/README.md`里的索引，再按问题类型查对应PDF。

## COMSOL 直连执行路径（默认）

当前项目默认使用`matlab.exe -batch`驱动的MATLAB LiveLink/Java API直连COMSOL，不把MCP当作正式执行路径。
标准形态是先单独启动`comsolmphserver.exe`，再由每个阶段脚本用`mphstart(2036)`连接同一个服务端，
然后按项目脚本里的专用判据完成加载、求解和后处理。

这条路径是当前项目的权威实现。涉及复杂已验证几何、项目专属检测判据或粒子到达时刻插值时，
仍以对应`ms_*.m`/`phase*.m`脚本为准，不要把任何“通用工具读取到的离散结果”替代专用判据。

如果后续某个会话里恰好还保留了`comsol` MCP，它也只适合作为临时检查手段，不作为默认工作流。

**本机连接状态（2026-07-13）**：MATLAB R2025b 位于
`C:\Program Files\MATLAB\R2025b\MatlabR2025b`。默认端口2036当前由PyCharm监听，不能再仅凭
“端口可连接”判定COMSOL Server就绪；测试应改用空闲端口并核对监听进程路径确为
`comsolmphserver.exe`。R2025b外层`matlab.exe -batch`可启动，但该模式下Java未启用；手工
`mphstart(2037)`虽建立TCP会话，120秒内未完成API初始化。当前已验证的自动路径是COMSOL
官方启动器`comsolmphserver matlab -mlroot <R2025b根目录>`，通过进程级`MATLABPATH`加载
`comsolstartup.m`：已在2037端口完成LiveLink 6.4.0.293连接、94.8 MB MPH加载、GUI节点检查、
`model.study('std1').run`重算和`mphinterp`后处理。正式生产脚本切换到R2025b前，仍需把这条
启动方式封装成项目入口，或继续解决外层`-batch`的Java问题。

## COMSOL 模型的 GUI 对等性

生成的 MPH 必须可由 COMSOL Desktop 完整接管：任何影响物理或数值结果的内容都必须作为
Model Builder 中可见、可编辑、可保存的节点或属性存在。范围包括几何、材料、网格、选择集、
参数/变量/函数/耦合算子、物理场及其域/边界/点条件、电势/电场/磁场、粒子释放和初始条件、
力、碰撞、壁面、Study、Solver、初值、解引用、数据集、派生值、绘图组和导出。

脚本只能作为创建这些模型节点的手段，不能把关键物理或数值逻辑留在 MATLAB 运行时、未持久化的
COMSOL Server 状态，或 GUI 无法编辑的外部代码中。验收时应重新打开生成的 MPH，确认相关设置
可检查和修改，并在对应 Study 的 GUI Compute 路径验证结果；仅验证`model.sol(...).runAll`不足。

## 目录结构

```text
simulation_repo/                    # 本 Git 仓库：只放可复现、可审阅资产
  README.md                          # 本文件
  COMSOL_MATLAB_API手册.md            # 通用 API 速查（跨部件）
  COMSOL_调试方法论.md                 # 通用调试方法论（跨部件）
  official_docs/                     # COMSOL 官方 PDF 与索引
  common/                            # 跨部件复用代码与长期验证基线
  components/
    mass_analyzers/oa_tof/dual_stage_ringstack/
    ion_guides/rf_quadrupole_collision_cooling/
    electron_sources/wehnelt_electron_gun/
    ion_sources/electron_impact/
```

大型模型和结果不进入本仓库，而放在工作区同级的`artifacts/components/`，其部件路径与
`simulation_repo/components/`镜像。以oa-TOF为例：正式MPH在
`artifacts/components/mass_analyzers/oa_tof/dual_stage_ringstack/models/comsol/formal/`，
历史模型在`archive/`，运行生成的中间模型在`scratch/comsol/`，结果在`results/comsol/`。
可复现的SIMION GEM/Lua/FLY2文本源在组件`simion/`；可由SIMION GUI直接打开的完整运行时
工作区在对应`artifacts/.../models/simion/workspace/`。新脚本必须通过组件路径解析函数
定位产物，不得再硬编码用户名，也不得写入旧根级`comsol_models/`、`comsol_results/`、
`cad_exports/`或`simion_oatof/`。

**判断一条新发现该写进哪个文件的经验法则**：
- 换个项目、换个部件还成立吗？→ 成立，是"怎么调用API"层面的 → 写进`API手册.md`。
- 换个项目、换个部件还成立吗？→ 成立，是"怎么排查问题"层面的策略/方法论 → 写进
  `调试方法论.md`。
- 只对这一个具体模型/这一套具体几何参数成立（比如"这个反射镜的d1应该取120mm"、"这个
  电子枪的Wehnelt孔径该多大"）→ 写进对应的`项目_*.md`。
- **同一个主题"表面上像API调用，其实是两种不同性质的知识"时，拆开分别归类**：例如
  画图——"怎么调API让标题变成这样"(`titletype='manual'`这类机械调用写法)属于API手册；
  "标题必须包含什么信息、颜色语义要不要跨图固定"这类**为什么要这么做**(防止review结果
  时被小样本噪声/图与图颜色语义不一致误导)的规范，本质是调试严谨性问题，应该写进
  `调试方法论.md`，不要因为表面上都叫"画图"就塞进同一份文档。判断标准依然是"这条知识
  换个项目是否还成立、成立的原因是API机制还是排查方法论"，跟其他分类判断同一套逻辑。
- 新开一个全新项目（不是给现有项目加功能）→ 新建一个`项目_<项目名>.md`，并在这份指南
  的表格里加一行。

## 各文档内部的维护规则

### `COMSOL_MATLAB_API手册.md`
- §7.0"已验证部件库速查表"和"7内容分类索引"是最高频被查阅的入口，新增一个完整部件
  时，两处都要同步加一行/一条，且要保持指向正确的小节号。
- §7.x的小节编号**只递增，不要在已有编号中间插入**——文档内外都有大量`见§7.x`交叉引用
  依赖编号稳定，插队会破坏引用。
- §7.12黑名单：确认某个调用无效/报错时顺手加一条，节省下次踩同一个坑的时间。
- 每条尽量只留"怎么写是对的"+一句话背景，详细的排查过程如果本身有通用方法论价值，放
  进`调试方法论.md`并互相`见xxx.md`引用；如果只是这一个项目的踩坑细节，放进对应的
  `项目_*.md`，这份API手册里不要堆砌叙事。

### `COMSOL_调试方法论.md`
- 只收录**已经被验证在不止一种场景下有效**的策略——如果只在一个项目里用过一次，先放
  进那个项目的`项目_*.md`里，等确认它在第二个不同场景下也有效之后，再"提炼"搬到这份
  方法论文件里。这样能避免这份文件被"其实是个例"的内容稀释。
- 每一条策略尽量写成"遇到X现象 → 用Y方法诊断 → 看到Z结果说明是A原因/继续查B"这种可
  直接照着执行的结构，不要只是叙述"我当时是怎么发现的"。

### `项目_*.md`
- 允许（且应该）比API手册/方法论文件更啰嗦、更叙事化——这里的目标是让下次接手这个项目
  的人（或AI）快速建立起"现在是什么状态、为什么是这个状态"的完整图景，而不是极简的
  速查表。
- 但**历史上被推翻/被更新公式覆盖的旧结论，应该整理成"最终结论+简要原因"，不需要保留
  完整的"试了A方案失败→试了B方案也失败→最后用C方案成功"这种逐帧过程**，除非那个失败
  过程本身包含值得记住的教训（比如"为什么直觉上合理的A方案会失败"）。
- 文件顶部应该有一个"当前参数/当前状态"速览区，以及"当前已知的开放问题"清单——这是
  下次会话最需要先看到的两块内容，其余按主题（不一定按时间顺序）组织。

## 何时应该新建文件 vs 加进现有文件

- 新项目（新的物理器件组合、新的目标应用，比如以后如果做磁扇形场质谱仪的完整项目而不
  只是组件测试）→ 新建`项目_<名称>.md`。
- 对现有项目的延续性工作（同一个模型改参数、修bug、加功能）→ 加进对应的`项目_*.md`，
  按主题归入合适的章节，不需要每次追加新章节到文件末尾。
- 发现一个新的、可复用的COMSOL组件/子系统（比如新的一种碰撞类型、新的一种物理场）→
  在`API手册.md`新开一个`§7.x`小节 + 更新§7.0表格。
- 发现一个新的、跨项目通用的排查策略 → 加进`调试方法论.md`对应主题的章节（不是每次都
  新建章节，先看现有的1-6节能不能放进去）。

## 本目录下的模型文件（.mph）与脚本

- 每个"项目"通常只保留**当前有效、最终版本**的建模脚本和对应.mph模型；被淘汰/被取代
  的中间版本脚本和模型应当删除，不要靠"注释里写着已废弃"来标记——具体见对应项目文件里
  "参考脚本"一节列出的当前有效脚本名单，那才是权威列表。
- 临时性的扫描/验证脚本（如`scan_d1.m`一次性用于寻找某个参数的最优值）如果其扫描结果
  已经被写进对应项目文档，脚本本身可以删除，不需要永久保留在目录里"以防万一"——文档里
  的表格/结论就是这次扫描的存档。

### 文件名前缀 = 生命周期约定（硬性规则）

| 前缀 | 生命周期 | 规则 |
|---|---|---|
| `scan_*.m` / `tmp_*.m` | **一次性**，用于寻找某个参数的最优值/做一次性对比 | 结果一旦写进对应`项目_*.md`的文档表格/结论，脚本本身和它产生的`.mph`/`.png`/`.log`**必须删除**，不允许"以防万一"长期保留。写这类脚本时脑子里就应该清楚它注定要被删。 |
| `test_*.m` | **永久保留**，独立组件的验证脚本 | 对应`COMSOL_MATLAB_API手册.md`§7.0部件库表格里的条目，任何项目都可能复用/参考，不要删。 |
| `ms_*.m` / `phase*.m` | **项目正式生产脚本**，长期维护 | 对应某个`项目_*.md`里"参考脚本"列出的当前有效版本。旧版本被新版本取代后才删除旧版本，正在用的不要动。 |

新建脚本时先决定它属于哪一类、按对应前缀命名，而不是事后才纠结"这个该不该删"。

## Git 管理规则

### 仓库边界

- Git 仓库根目录是`simulation_repo/`；提交只写入本机该目录下的`.git`历史。当前没有配置远程仓库，`git commit`不会上传文件。
- 仓库管理可复现的脚本、文档和必要的轻量理论资料；仓库外同级的`artifacts/`用于存放模型、CAD、计算结果和运行时工作区，不属于版本库。

### 哪些文件进入版本控制

| 类型 | Git 管理规则 |
|---|---|
| `CLAUDE.md`、本指南、`COMSOL_*.md`、`项目_*.md` | 必须跟踪。它们是项目知识、当前状态和操作约束的权威记录。 |
| `common/test_*.m` | 必须跟踪。它们是可复用组件的长期验证脚本。 |
| `components/**/ms_*.m`、`components/**/phase*.m` | 必须跟踪。它们是正式生产脚本，修改时同步更新对应项目文档。 |
| `scan_*.m`、`tmp_*.m` | 不作为长期版本历史保存。结论写入对应`项目_*.md`后删除脚本。 |
| `.mph`、图片、扫描数据、运行日志及其他计算输出 | 不提交。保存在工作区`artifacts/`；删除前仍须人工确认。 |
| `official_docs/`中的官方原始资料 | 不作为日常提交对象。仅在明确需要离线部署、版本锁定或版权审查时，再单独决定是否纳入。 |

### 提交原则

- 一次提交只表达一个可审阅主题，例如一次oa-TOF参数结论、一个通用组件修复，或一次文档整理；不要将不同项目的无关改动混在同一提交中。
- Agent 在一个主题完成并完成相称验证后应自行提交；任务包含多个独立主题时，应在切换主题前分别提交，无需等待人工催促。
- 自动提交只包含当前主题的改动。任务开始前已有的无关改动，或无法安全拆分的混合差异，不得夹带提交。
- 修改项目正式脚本时，必须同步更新对应`项目_*.md`，说明当前参数、结果或已知限制。
- 新增或修改跨项目可复用组件/API经验时，必须同步维护`COMSOL_MATLAB_API手册.md`中的§7.0和对应索引；跨项目验证的排查策略才写入`COMSOL_调试方法论.md`。
- 提交信息使用简短的祈使句，并建议采用范围前缀：`oatof:`、`egun:`、`common:`、`api:`、`docs:`、`chore:`。

### 暂存与提交前检查

不要习惯性使用`git add .`。使用明确文件路径或`git add -p`，避免把临时文件、模型或无关改动纳入提交。

提交前依次检查：

```powershell
git status --short --branch
git diff --check
git diff --stat
```

确认暂存区没有`.mph`、结果文件、日志或`scan_*.m`/`tmp_*.m`，并确认每项脚本变更已有匹配层级的文档记录。对已有未提交改动，先查看差异并按主题拆分；不得为了凑成一次提交而覆盖、回退或夹带其他工作。

仓库配置了版本控制的`.githooks/pre-commit`钩子；它会执行暂存区空白检查，并拒绝提交`.mph`、`official_docs/*.pdf`、`scan_*.m`和`tmp_*.m`。本地Git配置必须保持`core.hooksPath=.githooks`。

### 删除规则

- `scan_*.m`和`tmp_*.m`仅在对应结论已写入项目文档后删除。
- `.mph`和计算结果的删除必须先获得人工确认，即使它们不在Git仓库内。
- 正式`ms_*.m`/`phase*.m`仅在被新版本替代、对应项目文档已更新后删除。
- `test_*.m`默认保留；只有确认其不再是可复用验证基线，并同步更新API手册后才能删除。
